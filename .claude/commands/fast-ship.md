---
description: Fast ship cycle for small safe edits — same pipeline as /ship but auto-selects sensible defaults and skips local tests.
---

## When to use

Small, low-risk edits you are confident about: typo fixes, doc updates, one-line tweaks, config bumps. If the change touches logic, tests, or anything that could break CI use `/ship` instead.

## What fast-ship skips vs /ship

- No confirmation prompts between steps — runs straight through
- No local tests — CI is the safety net
- No commit-split proposal — single commit, stage everything
- No commit message approval — auto-generated from diff
- No PR option prompts — fixed defaults (base=main, not draft, auto title/body)
- No merge-method choice — always squash
- No branch-delete prompt — always delete

## What fast-ship keeps (non-negotiable)

- Lint — fast, catches obvious mistakes
- Conflict marker scan — never push broken merges
- Rebase if behind remote — always rebase, never merge
- CI wait — if CI fails, stop and surface. Do not auto-merge red CI.
- Branch hygiene — never push directly to main or master

## Steps

### 0. Git session lock

Acquire the git session lock before proceeding. See `_git-lock.md` for the full lock-check snippet.

Run the lock-check command. If LOCKED: stop and inform the user. If STALE: override. If FREE, acquire:

```powershell
"$([DateTime]::UtcNow.ToString('o')) /fast-ship" | Set-Content ".git/.claude-git-lock"
```

---

### 1. Clean-tree check + branch check + auto-create

```bash
git branch --show-current
git status --porcelain
git diff HEAD
```

If the working tree is clean (no uncommitted changes, user pre-committed):
- If on a feature branch: skip to Step 3 (push). The commit already exists.
- If on main or master: stop — tell the user they committed directly to main. They need to create a branch and cherry-pick, or use `/pr` to handle it.

If there are uncommitted changes, analyze the diff first to determine the Conventional Commit type (feat/fix/docs/chore/etc).

If on main or master, auto-create a feature branch based on the diff analysis:
- `fix/<short-description>` for fix type commits
- `feat/<short-description>` for feat type commits
- `docs/<short-description>` for docs type commits
- `chore/<short-description>` for everything else

Derive the short-description from the diff content (kebab-case, max 40 chars).

```bash
git checkout -b <auto-branch-name>
```

If already on a feature branch: use it as-is.

---

### 2. Stage + lint + commit

Stage tracked changes and new source files:

```bash
git add -u
git status --porcelain | grep "^??" | grep -vE "\.(env|log|tmp|bak|swp)$"
```

Review the untracked files list. If any look like they belong in the commit, `git add <file>` them. Otherwise leave them unstaged.

```bash
git diff --staged
```

Run lint (fast, non-negotiable):

```bash
ruff check .
```

If lint fails: stop, surface the error. Do not commit.

Scan for conflict markers:

```bash
git diff --staged | grep -E "^\+(<<<<<<<|=======|>>>>>>>)"
```

If conflict markers found: stop, surface them.

Auto-generate a Conventional Commit message from the diff (same rules as `/commit` Step 4 — type, scope, imperative mood, 72 chars max). No approval prompt.

```bash
git commit -m "<auto-generated message>"
```

---

### 3. Remote sync + push

Run lint (in case Step 2 was skipped via the clean-tree path):

```bash
ruff check .
```

If lint fails: stop, surface the error. Do not push.

```bash
git fetch origin
git ls-remote --exit-code origin <current-branch>
```

If ls-remote fails (new branch): skip sync, use --set-upstream on push.

If ls-remote succeeds, check if behind:

```bash
git rev-list --count HEAD..origin/<current-branch>
```

If behind by more than 0: auto-rebase (no prompt):

```bash
git rebase origin/<current-branch>
```

If rebase has conflicts: stop, surface the conflict. Do not force anything.

Push:

```bash
git push --set-upstream origin <current-branch>
```

Or after rebase:

```bash
git push --set-upstream --force-with-lease origin <current-branch>
```

---

### 4. Create PR

Write the body to a temp file to avoid shell escaping issues:

```powershell
"<auto-body>" | Set-Content -Path "$env:TEMP\pr-body.md" -Encoding utf8
gh pr create --base main --title "<auto-title>" --body-file "$env:TEMP\pr-body.md"
Remove-Item "$env:TEMP\pr-body.md"
```

- Title: derived from the commit message
- Body: auto-generated from commit log, minimal
- Not draft — ready for review and merge immediately

Capture the PR number and URL from output.

---

### 5. Wait for CI

Poll until all checks complete:

```bash
gh pr view <PR-NUMBER> --json statusCheckRollup --jq ".statusCheckRollup[] | {name, status, conclusion}"
```

If any checks are pending: wait 10 seconds, re-check. Show which checks are still running.

If all checks passed: proceed to Step 6.

If any checks failed: stop and surface the failures. Do not auto-merge.

Tell the user: CI failed. Fix the issue and re-run `/fast-ship`, or merge manually with `/pr`.

---

### 6. Merge (squash, delete branch)

```bash
gh pr merge <PR-NUMBER> --squash --delete-branch
```

Post-merge cleanup:

```bash
git checkout main
git pull origin main
git branch -D <merged-branch-name>
```

---

### 7. Done

Show a compact summary:

- PR number and URL
- Merge method: squash
- Branch: deleted
- Current branch: main (up to date)

If you are ready to release, run `/publish`.

---

## Cleanup

On any exit — whether the command completes successfully, the user aborts, or an error occurs — always release the git session lock before ending:

```powershell
Remove-Item ".git/.claude-git-lock" -ErrorAction SilentlyContinue
```
